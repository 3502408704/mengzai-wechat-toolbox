# 排版助手 Phase 1 设计文档

- 日期：2026-05-21
- 项目名：排版助手
- 平台：Android 8.0+（API 26+）
- 设计目标：以 Kotlin 全量重构现有原型，交付一个稳定、美观、符合 Material 3 与无障碍规范的 Android APK，让用户能够导入、编写、预览并复制适合粘贴到微信公众号编辑器等第三方应用中的排版内容。

## 1. 范围与阶段

本设计文档覆盖 `Phase 1`，重点完成本地排版核心链路，不在本阶段接入真实 AI API。应用主链路为“导入或输入内容 -> 安全预览 -> 复制成品 -> 粘贴到第三方编辑器”，并补齐草稿保存、历史记录、导入导出、主题、无障碍和实验室占位等基础能力。

### 1.1 Phase 1 包含

- Kotlin 全量替换现有 Java 原型
- Jetpack Compose + Material 3 界面重建
- HTML、CSS、Markdown 内容输入与编辑
- 内容类型识别与统一预览渲染
- 安全清洗与受控 WebView 预览
- 复制预览成品到系统剪贴板，附带 HTML 与纯文本
- 系统剪贴板内容识别与一键导入
- 文件导入：`.html`、`.md`、`.txt`
- 文件导出：`.html`、`.txt`
- 系统分享
- 自动保存当前草稿
- 历史快照列表、恢复、重命名、删除、收藏
- 深浅色模式与 Android 12+ Dynamic Color
- 基础代码编辑增强：等宽字体、行号开关、字号调节、Markdown 快捷栏、撤销/重做
- 首次使用引导
- 开发者模式与基础调试信息入口
- 无障碍支持
- `AI 辅助` 底部导航占位与设置页实验室条目占位

### 1.2 Phase 1 不包含

- 真实 AI 对话能力
- 通用 API Key、Base URL、Model 配置页
- AI 生成结果导入编辑器
- 编辑页与预览页实时分屏
- 专业级代码编辑器能力，如完整语法树、高级补全、折叠等

### 1.3 Phase 2 预告

第二阶段将基于原生本地 UI 扩展 `AI 辅助` 模块：

- 原生聊天界面
- 通用 OpenAI 兼容接口配置
- 自定义 `Base URL / API Key / Model`
- AI 生成 HTML、CSS、Markdown 代码提取
- 一键导入编辑器并进入预览

Phase 2 不采用网站嵌入或 WebView 接站方案，而采用原生 UI 与可配置 API 的实现路径。

## 2. 产品目标

### 2.1 用户目标

- 快速把现成 HTML、CSS、Markdown 排版内容导入手机
- 在手机上完成基础修改与安全预览
- 一键复制成品，尽量减少粘贴到公众号编辑器后的样式丢失
- 不因误操作、崩溃或切换页面丢失内容
- 在未来版本中，通过原生 AI 辅助页获得更顺滑的生成体验

### 2.2 产品成功标准

- 用户从启动应用到复制预览成品的主路径顺畅且无需学习成本
- 粘贴到常见公众号编辑器时，能保留主要文本结构、层级、强调、列表和基础样式
- 历史恢复、导入导出、主题切换和无障碍流程均可独立使用
- 在 Android 8.0+ 设备上保持可用并具备合理性能
- `AI 辅助` 入口在 Phase 1 中具备清晰预期管理，不误导用户为已可用功能

## 3. 技术路线

采用原生主导型架构：

- 语言：Kotlin
- UI：Jetpack Compose + Material 3
- 导航：Navigation Compose
- 存储：Room + DataStore
- 预览渲染：受控 Android WebView
- 依赖注入：Hilt
- 异步：Kotlin Coroutines + Flow
- 文件导入导出：Storage Access Framework

选择该路线的原因：

- 最适合统一 Material You、无障碍与页面状态管理
- 历史记录、设置、剪贴板导入等本地能力更容易稳定集成
- 预览和复制链路适合用本地数据流统一管理
- 后续增加原生 `AI 辅助` 子系统时，可直接复用导航、主题和数据层

## 4. 信息架构与页面结构

### 4.1 顶层结构

应用采用单 Activity + 多页面导航结构：

- `EditorScreen`
- `PreviewScreen`
- `HistoryScreen`
- `AiAssistantScreen`
- `SettingsScreen`
- `OnboardingScreen`

底部导航包含：

- 编辑
- 预览
- 历史
- AI 辅助
- 设置

其中：

- `编辑`、`预览`、`历史`、`设置` 为 Phase 1 可用功能
- `AI 辅助` 为 Phase 1 灰态占位，不可进入真实功能流程

### 4.2 编辑页

编辑页包含以下区域：

- 顶栏
  - 导入文件
  - 撤销
  - 重做
- 编辑器控制区
  - 行号显示开关
  - 字号调节
  - Markdown 快捷工具栏
- 主编辑器卡片
  - 等宽字体大文本输入区
  - 可选行号列
  - 基础语法高亮
- 剪贴板识别提示条
- 底部辅助操作
  - 复制源码
  - 新建或清空
  - 前往预览

设计原则：

- 保持单屏主任务聚焦于输入与修改
- 编辑页负责“写”和“改”，不承担最终成品复制职责
- 预览入口明显，但不把清洗、导出、分享逻辑塞进编辑器主区域

### 4.3 预览页

预览页作为底部导航正式主页面存在，包含：

- 顶栏
  - 刷新预览
  - 导出
  - 分享
- 内容区域
  - 受控 WebView 渲染结果
  - 加载中、空状态、渲染失败状态
- 底部固定主按钮
  - 复制预览内容到剪贴板

预览页负责：

- 展示清洗后的最终结果
- 提供复制、导出、分享
- 处理 WebView 失败与刷新
- 支持上下滚动与 WebView 受控缩放

### 4.4 历史页

历史页展示自动保存快照列表，每条记录至少包含：

- 标题摘要
- 内容类型
- 更新时间
- 收藏状态

支持以下操作：

- 恢复
- 重命名
- 删除
- 收藏或取消收藏

### 4.5 AI 辅助页

`AI 辅助` 在 Phase 1 中作为底部导航占位页面存在，页面状态为：

- 入口可见
- 默认灰态或禁用态视觉
- 点击后展示“实验室功能，后续版本开放”的说明
- 不展示可误导为真实可用的聊天输入框或厂商品牌入口

该页面的职责是：

- 提前建立产品结构预期
- 为后续 Phase 2 功能扩展保留导航位
- 避免未来上线 AI 模块时再重构底部信息架构

### 4.6 设置页

设置页包含：

- 主题模式：浅色、深色、跟随系统
- Dynamic Color 开关
- 默认字号
- 行号默认开关
- 开发者模式
- 实验室功能分组
  - `AI 辅助编辑（即将推出）`
- 清理缓存
- 关于与隐私说明

## 5. 核心交互流程

### 5.1 主链路

1. 用户进入编辑页
2. 应用在前台检测剪贴板内容是否疑似 HTML、CSS、Markdown
3. 若命中规则，弹出提示：“检测到排版内容，是否导入到编辑器？”
4. 用户编辑内容，应用节流自动保存当前工作草稿
5. 用户切换到预览页或点击前往预览
6. 应用识别内容类型，生成最终安全 HTML
7. 在预览页 WebView 中渲染
8. 用户点击复制预览内容到剪贴板
9. 应用同时写入 HTML 与纯文本，并通过 Snackbar 提示复制成功

### 5.2 导入流程

支持以下导入方式：

- 剪贴板导入
- 文件导入
- 手动输入或粘贴

所有导入内容统一进入当前工作草稿，再由用户决定是否预览、导出或复制。

### 5.3 历史恢复流程

恢复历史记录时不直接覆盖现有工作草稿，而是：

1. 先为当前内容生成快照
2. 再载入所选历史内容作为新的工作草稿

该策略可以最大限度避免误恢复导致的内容丢失。

### 5.4 AI 占位交互

用户进入 `AI 辅助` 页时：

1. 显示实验室卡片
2. 说明该功能将在后续版本开放
3. 可引导用户前往设置页查看实验室说明

该流程不触发网络、不请求 API、不展示未实现配置项。

## 6. 内容识别与预览生成

### 6.1 输入类型识别

使用轻规则识别，不追求编译器级严格判断：

- 基本无 HTML 标签，且包含标题、列表、链接、强调等 Markdown 特征时，按 Markdown 处理
- 出现明显 HTML 标签时，按 HTML 处理
- 若出现 `<style>` 标签或明显 CSS 规则，提取为样式层
- 混合内容时，优先保留 HTML 结构；Markdown 片段先转 HTML，再拼接到正文
- 无法识别时按普通文本处理，并转义后包裹为段落文本

### 6.2 生成流程

统一生成最终预览 HTML：

1. 识别内容类型
2. Markdown 内容转换为 HTML
3. 提取和整理 CSS
4. 组装文档骨架
5. 注入移动端 `meta viewport`
6. 执行安全清洗
7. 输出给 WebView、复制、导出与分享模块

### 6.3 预览文档骨架

输出的 HTML 文档至少包含：

- `<!DOCTYPE html>`
- `html/head/body`
- `meta charset="utf-8"`
- `meta name="viewport" content="width=device-width, initial-scale=1.0, maximum-scale=1.0"`
- 基础阅读样式
- 用户清洗后的安全样式
- 渲染正文容器

## 7. 安全模型

预览必须默认安全，不能信任原始输入。

### 7.1 清洗规则

默认移除或过滤：

- `script`
- `iframe`
- `object`
- `embed`
- `form`
- `input`
- `button`
- 任何内联事件属性，如 `onclick`、`onload`
- 危险协议，如 `javascript:`

默认保留的常见排版标签包括但不限于：

- `p`
- `div`
- `span`
- `h1` 到 `h6`
- `strong`
- `em`
- `blockquote`
- `ul`
- `ol`
- `li`
- `img`
- `a`
- `pre`
- `code`
- `table`
- `thead`
- `tbody`
- `tr`
- `th`
- `td`

### 7.2 CSS 策略

首版允许基础排版样式，但限制高风险能力：

- 允许文字、颜色、边距、内边距、边框、背景、对齐、行高、字号、宽度等排版相关样式
- 限制或剔除极端定位、行为脚本相关样式、危险表达式与已废弃的可执行能力
- 对图片、表格等元素附加移动端自适应兜底样式，防止横向溢出

### 7.3 WebView 约束

- 禁止任意跳转离开应用主流程
- 禁止加载不可信远程脚本
- 禁止文件系统级敏感访问
- 仅将 WebView 用作渲染容器，不作为通用浏览器

## 8. 编辑器设计

首版使用 Compose 原生增强型编辑器，而不是外接重型代码编辑器。

### 8.1 首版能力

- 大文本输入
- 等宽字体
- 行号显示开关
- 基础语法高亮
  - 标签
  - 属性
  - 普通文本
- 撤销和重做
- 字号调节
- Markdown 快捷工具栏
  - 标题
  - 加粗
  - 斜体
  - 引用
  - 列表
  - 链接

### 8.2 非目标

首版不做：

- 自动补全
- 折叠代码
- 语义诊断
- 多光标

该取舍用于换取整体稳定性和较低维护成本。

## 9. 剪贴板识别

### 9.1 触发时机

仅在以下时机检测：

- 应用启动进入前台
- 应用从后台回到前台

不在后台静默读取，不上传任何剪贴板数据。

### 9.2 识别规则

识别内容是否可能为排版内容的启发式特征：

- 包含 HTML 标签
- 包含 CSS 选择器与规则块
- 包含常见 Markdown 语法，如标题、列表、链接、代码块

命中后弹出底部提示条，提供：

- 粘贴到编辑器
- 忽略

### 9.3 兼容策略

- 遵循 Android 各版本前台读取行为
- 若系统限制导致内容不可用，则不报错打断主流程，只不展示导入提示

## 10. 数据模型与存储

### 10.1 当前工作草稿

当前工作草稿始终只有一份，保存：

- 原始内容
- 当前标题
- 推断内容类型
- 最近预览 HTML
- 最近编辑时间

输入后通过节流自动保存，避免频繁写盘。

### 10.2 历史快照

在以下节点创建历史快照：

- 预览成功后
- 导入内容后
- 手动新建前
- 从历史恢复前

每条历史记录保存：

- `id`
- `title`
- `rawContent`
- `lastRenderedHtml`
- `contentType`
- `isFavorite`
- `createdAt`
- `updatedAt`

### 10.3 容量策略

- 默认最多保留最近 100 条历史记录
- 超出后自动淘汰最旧且未收藏的记录
- 收藏记录不参与自动淘汰，除非用户主动删除

## 11. 文件导入导出与分享

### 11.1 导入

支持导入：

- `.html`
- `.md`
- `.txt`

优先按 UTF-8 读取，若失败则尝试常见编码回退。

### 11.2 导出

支持导出：

- `.html`
- `.txt`

`.html` 导出为完整文档，包含：

- viewport
- 清洗后的结构
- 安全样式

`.txt` 导出为纯文本阅读内容。

### 11.3 分享

通过系统 `ACTION_SEND` 分享：

- 导出的文件
- 或可分享的纯文本内容

## 12. 复制到剪贴板

复制预览成品时同时写入：

- HTML 表示
- 纯文本表示

目标是尽量兼容公众号编辑器等第三方应用的粘贴行为。若目标应用无法完整识别 HTML，至少应保住纯文本内容不丢失。

复制成功后使用 Snackbar 给出非阻塞提示，并保证无障碍服务可感知。

## 13. 主题、视觉与品牌

### 13.1 主题系统

- 使用 Material 3
- 支持浅色、深色、跟随系统
- Android 12+ 支持 Dynamic Color
- Android 8.0 到 11 使用预设色板回退

### 13.2 视觉原则

- 界面干净、层级清晰，突出编辑与预览主任务
- 编辑器使用代码风格配色，但不做“开发工具”式复杂视觉负担
- 预览页使用阅读友好的容器背景与间距
- AI 占位页使用低干扰说明样式，避免造成“功能已可用”的误解

### 13.3 代码主题

- 浅色模式下提高文本清晰度和分区感
- 深色模式下提升标签、属性、文本的颜色对比，避免发灰难辨认

## 14. 无障碍要求

所有可交互控件必须具备明确的无障碍语义。

### 14.1 组件要求

- 按钮使用 `Button` 或语义化按钮角色
- 开关使用 `Switch`
- 单选使用 `RadioButton`
- 复选使用 `Checkbox`
- 图标按钮补充明确的 `contentDescription`

### 14.2 交互要求

- 触控目标最小 48dp
- 焦点顺序明确且符合阅读逻辑
- 编辑器、提示条、复制结果、错误状态可被 TalkBack 感知
- 不能仅依赖颜色表达成功或失败状态
- 文本与背景对比度满足 WCAG AA
- 灰态 AI 入口仍需可被读屏明确读出为“即将推出，不可用”

### 14.3 重点文案示例

- “预览按钮”
- “导入文件”
- “撤销编辑”
- “复制预览内容到剪贴板”
- “切换行号显示”
- “选择浅色主题”
- “AI 辅助，即将推出”

## 15. 首次使用引导

首次启动显示简洁引导，介绍：

- 可以粘贴或输入 HTML、CSS、Markdown
- 可以在预览页复制排版结果
- 可以从剪贴板和文件一键导入
- 可以在历史中恢复之前的内容
- AI 辅助将在后续实验室版本开放

引导应尽量轻量，支持跳过，完成后不再强制展示。

## 16. 性能与兼容性

### 16.1 目标

- Android 8.0+ 可运行
- 大段文本编辑保持基本流畅
- 预览生成与切换等待时间可接受

### 16.2 兼容与降级

- Dynamic Color 不可用时自动使用默认主题
- WebView 异常时展示刷新与错误提示
- 某些第三方应用粘贴 HTML 不稳定时保留纯文本兜底
- AI 辅助在 Phase 1 中不触发任何未实现网络逻辑

## 17. 开发者模式

设置页提供开发者模式，打开后可查看基础调试信息：

- 最近一次内容识别类型
- 最近一次预览生成时间
- 最近一次 WebView 渲染错误摘要

首版不直接暴露复杂控制台，只提供排查所需的轻量信息。

## 18. 测试策略

### 18.1 单元测试

覆盖以下逻辑：

- 内容类型识别
- Markdown 转 HTML
- 安全清洗
- 历史淘汰策略
- 剪贴板特征识别

### 18.2 集成测试

覆盖以下流程：

- 导入文件
- 生成预览 HTML
- 复制到剪贴板
- 从历史恢复
- AI 占位页禁用状态展示

### 18.3 手工验收

- 深浅色切换正确
- TalkBack 基本可读
- 长文本编辑不卡顿或可接受
- 导出与分享链路可用
- 将复制结果粘贴到目标编辑器后结构基本稳定
- AI 辅助入口显示为灰态且说明准确

## 19. 后续阶段预留

Phase 1 完成后，第二阶段可扩展：

- 原生 `AI 辅助` 聊天页
- 可自定义 API 配置
- 代码块识别与导入编辑器
- AI 结果直接进入预览
- 全屏预览增强与更完整的缩放检查
- 可选分屏预览

为避免未来返工，Phase 1 会预留以下结构：

- `AiAssistantScreen` 导航入口
- AI 模块状态占位
- 设置页实验室分组
- 后续 API 配置的数据结构与入口位置

## 20. 风险与结论

### 20.1 主要风险

- 不同公众号编辑器对 HTML 粘贴支持存在差异，无法保证样式完全一致
- 原生编辑器的语法高亮能力首版只能做到够用，不等于专业代码编辑器
- WebView 对复杂 CSS 的渲染表现可能与目标编辑器不同
- Phase 1 中 AI 入口可见但不可用，需通过文案和视觉准确管理用户预期

### 20.2 结论

本方案以“稳定完成移动端排版复制主链路”为优先目标，使用 Kotlin 与 Compose 重构现有原型，先把本地编辑、预览、安全、历史与导出体验做扎实。同时保留 `AI 辅助` 的产品结构占位，但明确将真实能力放到后续以原生 UI + 可自定义 API 配置的方案实现，从而兼顾当前交付质量与后续扩展效率。
